## Overview

Sprint 1 delivers the foundational data layer and CLI for agile-backlog. After this sprint, users can create, list, filter, and move backlog items via the `agile-backlog` command, with all data stored as git-tracked YAML files.

## Design Decisions

### Backlog directory location â€” Git root auto-detection

The `backlog/` directory lives at the repository root, discovered via `git rev-parse --show-toplevel`. This means `agile-backlog` commands work from any subdirectory in the project.

If not inside a git repo, the CLI exits with a clear error message.

### ID derivation

Item IDs are derived from the filename: `backlog/fix-auth-leak.yaml` â†’ `id: fix-auth-leak`. The `id` field in the YAML is authoritative but must match the filename. On write, the filename is generated by slugifying the ID.

### Slug generation

IDs are created by slugifying the title: lowercase, replace spaces/special chars with hyphens, strip leading/trailing hyphens, collapse consecutive hyphens. If a slug collision occurs, append `-2`, `-3`, etc.

## Components

### 1. `src/models.py` â€” BacklogItem Pydantic model

```python
class BacklogItem(BaseModel):
    id: str
    title: str
    status: Literal["backlog", "doing", "done"] = "backlog"
    priority: Literal["P1", "P2", "P3"]
    category: str
    sprint_target: int | None = None
    created: date
    updated: date
    depends_on: list[str] = []
    tags: list[str] = []
    description: str = ""
    acceptance_criteria: list[str] = []
    notes: str = ""
```

- Validation: status/priority constrained by Literal types
- `created` and `updated` default to today on creation
- Provides `to_yaml_dict()` method that excludes `id` (derived from filename)
- Provides `slugify(title)` static method for ID generation

### 2. `src/yaml_store.py` â€” YAML file operations

**Functions:**

| Function | Purpose |
|----------|---------|
| `get_backlog_dir()` | Find git root, return `<root>/backlog/` Path, create if missing |
| `load_all()` | Read all `.yaml` files from backlog dir, return `list[BacklogItem]` |
| `load_item(item_id)` | Read single item by ID |
| `save_item(item)` | Write item to `backlog/<id>.yaml`, update `updated` date |
| `item_exists(item_id)` | Check if an item file exists |

**Behavior:**
- Uses `git rev-parse --show-toplevel` via `subprocess.run` to find repo root
- YAML files are read/written with PyYAML (`yaml.safe_load` / `yaml.dump`)
- Invalid YAML files are skipped with a warning (don't crash on one bad file)
- `backlog/` directory is created on first write if it doesn't exist

### 3. `src/cli.py` â€” Click CLI

**Commands:**

| Command | Description |
|---------|-------------|
| `agile-backlog list` | List all items (table format). Flags: `--status`, `--priority`, `--category`, `--sprint` |
| `agile-backlog add "title" --priority P2 --category feature` | Create new item |
| `agile-backlog move <id> --status doing` | Change item status |
| `agile-backlog show <id>` | Show full item details |
| `agile-backlog serve` | Placeholder â€” prints "Coming in Sprint 2" |

**Output format:**
- `list`: tabular output with columns: ID, Title, Status, Priority, Category
- `show`: full YAML-like detail view
- `add`/`move`: confirmation message with item ID
